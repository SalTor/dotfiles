# post-feedback - Post curated AI feedback to a PR/MR

Post feedback to a pull/merge request as inline review comments anchored to file:line **plus** a top-level summary. The full attribution line goes on the top-level summary only; each inline comment carries a bare `:robot:` marker so it's visibly machine-generated without repeating the attribution everywhere.

This is the write-side complement to `/check-feedback` (which only *reads* feedback). The content here is feedback you already have — typically the output of `/review` or `/code-review`, or text the user supplies. The "curated by Sal" attribution means a human signed off, so **always show the exact comments and get explicit confirmation before posting.**

## The attribution prefixes

The **top-level summary** (review body / MR note) must begin with this exact line, followed by a blank line:

```text
:robot: Generated by Claude, curated by Sal

<the summary body>
```

Each **inline comment** begins with a type tag, then the emoji token and a space, all on the same line as the start of the body:

```text
[<TYPE>] :robot: <the inline comment body>
```

Use these verbatim. Do not paraphrase, translate, or drop the emoji. The full "Generated by Claude, curated by Sal" line appears **only** on the top-level summary, never on inline comments. The type tag (next section) leads the inline line; the summary takes no type tag.

## Comment types (tag)

Every inline comment must open with exactly one bracketed, uppercase type tag so the reader knows at a glance whether it blocks, merely suggests, or is just context. Pick the one that fits — when torn between two, the more actionable one wins (`[ISSUE]` over `[SUGGESTION]`, `[SUGGESTION]` over `[NIT]`).

| Tag | Use when | Blocks merge? |
| --- | --- | --- |
| `[ISSUE]` | A concrete failure mode you can name — bug, broken invariant, security hole, regression. State what goes wrong and when. | Usually |
| `[SUGGESTION]` | A real improvement worth considering; author can take it or explain why not. | No |
| `[QUESTION]` | You need information to judge the code; you're asking, not asserting. Don't dress an assertion up as a question. | No (but may surface one) |
| `[NIT]` | Minor/cosmetic — naming, formatting, a stray word. Explicitly non-blocking. | No |
| `[NOTE]` | Observation or context with no required action. | No |
| `[PRAISE]` | A genuine positive callout. Use sparingly. | No |

Rules:

- **One tag per comment.** If a comment needs two tags, it's two comments — split it.
- **Don't inflate.** `[ISSUE]` is for nameable failure modes, not "I'd prefer". If you can't say what breaks and when, it's a `[SUGGESTION]`, `[QUESTION]`, or `[NOTE]`.
- The tag is the reader's triage signal — a wall of `[ISSUE]` tags on cosmetic points trains people to ignore them.

## Comment style: headline + bullets

Feedback often arrives as dense review prose (e.g. from `/review`). **Do not post it as-is.** Reshape every comment — inline and summary — into a skimmable structure before showing it for confirmation:

1. **Headline first.** One bold sentence stating the takeaway or ask. A reader who stops here should know what the comment is about and whether it needs action.
2. **Then 2–4 terse bullets** carrying the supporting evidence: what was checked, what's missing, the concrete suggestion. One idea per bullet. Use backticked `file/symbol/test-name` references, not prose descriptions of them.
3. **Optional one-line coda** for the "why it matters" framing, if it isn't already obvious from the headline.

Rules of thumb:

- Keep the substance, cut the connective tissue. Subordinate clauses ("Given that…", "…which means that…") usually become their own bullet or get dropped.
- A long verification list ("Verified: A; B; C; D; E") becomes a bulleted list, one item per line.
- Aim for roughly half the length of the source prose. If a comment still exceeds ~6 bullets, it's probably two comments — split it or demote the tail.
- The tag and emoji stay on the headline line: `[SUGGESTION] :robot: **headline…**`.
- **Avoid the tic phrases "load-bearing" and "blast radius"** — they're overused. Say the specific thing instead:
  - *blast radius* → "what this affects", "scope of impact", "how far the change reaches", "what's downstream", "what breaks if it's wrong".
  - *load-bearing* → "the code the PR hinges on", "on the critical path", "what depends on this", "central to X", "this carries the behavior".
  - Prefer naming the actual files/callers/paths over either metaphor.

Example transformation:

```text
Before (one blob):
[SUGGESTION] :robot: The v3 branch here — the /api/v3/... URL, the response
remap, and the null-body fallback — isn't covered by an integration test.
flattenV3.test.ts exercises the flatten function in isolation, but
DeliveryPromiseApi.server.test.ts already has a fetch-mock harness that
drives the full fetch→parse→group→filter path against v2. Given the
harness-first workflow in CLAUDE.md, consider adding a sibling v3 case ...

After:
[SUGGESTION] :robot: **The v3 branch (URL, response remap, null-body fallback) has no integration test.**

- `flattenV3.test.ts` covers the flatten fn in isolation only
- `DeliveryPromiseApi.server.test.ts` already has a fetch-mock harness driving the full fetch→parse→group→filter path — for v2
- Suggest a sibling v3 case there: mock a keyed-by-variant body, assert the `/api/v3/` URL, assert priority survives through `getByType`

This locks in the wiring the PR hinges on (harness-first workflow, CLAUDE.md).
```

## Usage

```text
/post-feedback [revset|number]
```

Examples:
- `/post-feedback` → infer the PR/MR from the bookmark on `@` (or nearest ancestor with a tracked bookmark)
- `/post-feedback @-` → use the bookmark on the parent change
- `/post-feedback 1234` → post to PR/MR number `1234` directly

If the feedback itself isn't already in the conversation, ask the user for it (or run `/review` first).

## Workflow

1. **Detect forge**
   ```bash
   git remote get-url origin
   ```
   - GitHub host → use `gh`.
   - GitLab host → use `glab`.
   - Verify auth before posting:
     ```bash
     gh auth status   # or: glab auth status
     ```

2. **Resolve the PR/MR number** (same resolution as `/check-feedback`)
   - If the user passed a bare number, use it.
   - Otherwise resolve from the revset:
     ```bash
     jj log -r '<rev>' --no-graph -T 'bookmarks ++ "\n"'
     ```
     - If no bookmark on the rev, walk back to the nearest ancestor with one:
       ```bash
       jj log -r '::<rev> & bookmarks()' --no-graph -T 'change_id.short() ++ " " ++ bookmarks ++ "\n"'
       ```
     - Map the bookmark to a PR/MR:
       ```bash
       gh pr list --head <bookmark> --state all --json number,url,state,title
       # or
       glab mr list --source-branch <bookmark> --all
       ```
   - If nothing matches, report that no PR/MR exists and stop. Don't guess.

3. **Anchor inline comments to the current head**
   - Inline comments must point at lines that exist in the PR's head commit, not stale local lines.
   - Get the head SHA the PR is currently showing:
     ```bash
     gh pr view <number> --json headRefOid -q .headRefOid
     ```
   - For each inline finding, capture `path`, the line number in the head version of the file, and `side` (`RIGHT` for added/context lines, `LEFT` for removed lines). Use `start_line`+`line` for multi-line ranges.
   - If a finding's line no longer exists at head, demote it into the summary instead of anchoring it — a bad anchor makes the API reject the whole review.

4. **Reshape and apply the prefixes**
   - Restructure every comment per "Comment style: headline + bullets" above — never post source prose as one blob.
   - Prepend the full attribution line + blank line to the summary body.
   - Pick one type tag per inline comment (see "Comment types") and lead with it: `[<TYPE>] :robot: ` (tag, emoji token + space, same line). No exceptions. The summary carries the attribution line, not a type tag.

5. **Confirm before posting** (outward-facing — required)
   - Show the user the full set: the summary body and each inline comment with its file:line anchor and type tag, all with the prefix applied.
   - Get explicit go-ahead. Do not post on inference.

6. **Post — GitHub (`gh`)**

   Post inline comments and the summary as a single review (atomic; groups everything under one submission). Build a JSON payload and pipe it to `gh api` — array fields via `-f`/`-F` flags are fragile, JSON input is reliable. `{owner}`/`{repo}` are auto-substituted by `gh`.

   ```bash
   gh api --method POST repos/{owner}/{repo}/pulls/<number>/reviews --input - <<'JSON'
   {
     "commit_id": "<headRefOid>",
     "event": "COMMENT",
     "body": ":robot: Generated by Claude, curated by Sal\n\n<summary text>",
     "comments": [
       {
         "path": "src/foo.ts",
         "line": 42,
         "side": "RIGHT",
         "body": "[SUGGESTION] :robot: <inline finding>"
       }
     ]
   }
   JSON
   ```

   - `event: "COMMENT"` posts feedback without an approval decision. Use `REQUEST_CHANGES` or `APPROVE` only if the user explicitly asks.
   - If the user instead wants the summary in the main conversation thread (not as the review body), post it separately:
     ```bash
     gh pr comment <number> --body ":robot: Generated by Claude, curated by Sal

   <summary text>"
     ```

7. **Post — GitLab (`glab`)**
   - Summary as a note on the MR:
     ```bash
     glab mr note <iid> --message ":robot: Generated by Claude, curated by Sal

   <summary text>"
     ```
   - Inline comments are positional discussions and need full diff-position context. Fetch the SHAs and post one discussion per line via the API:
     ```bash
     glab api projects/:fullpath/merge_requests/<iid> --jq '{base: .diff_refs.base_sha, start: .diff_refs.start_sha, head: .diff_refs.head_sha}'

     glab api --method POST projects/:fullpath/merge_requests/<iid>/discussions \
       -f 'body=[SUGGESTION] :robot: <inline finding>' \
       -f 'position[position_type]=text' \
       -f 'position[base_sha]=<base_sha>' \
       -f 'position[start_sha]=<start_sha>' \
       -f 'position[head_sha]=<head_sha>' \
       -f 'position[new_path]=src/foo.ts' \
       -f 'position[new_line]=42'
     ```

8. **Report what was posted**
   - Link to the posted review/comment(s). For GitHub the review URL comes back as `html_url` in the API response; otherwise point at `gh pr view <number> --web`.
   - State how many inline comments landed and whether any findings were demoted to the summary (per step 3).

## Notes

- **The prefixes are non-negotiable** — full attribution line on the top-level summary, `[<TYPE>] :robot: ` on every inline comment. If you can't apply the right prefix and a valid type tag to a given comment, don't post that comment.
- **Confirm before posting.** This writes to a PR other people see; treat it like sending mail. Approval to post one batch does not extend to a later one.
- A single bad inline anchor (line that doesn't exist at head) rejects the *entire* review on GitHub. Verify anchors against `headRefOid`, and demote anything stale into the summary.
- Multi-line GitHub comments: include both `start_line` and `line` (and matching `side`/`start_side`).
- Use the review API (`/pulls/<n>/reviews`) to post inline + summary atomically rather than firing many `/pulls/<n>/comments` calls.
- Cite **Jujutsu change IDs** for local changes and **PR/MR URLs** for posted feedback so the user can click straight to either side.
- Git is a fallback only if `jj` is unavailable in the repo.
