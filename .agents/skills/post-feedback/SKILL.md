# post-feedback - Post curated AI feedback to a PR/MR

Post feedback to a pull/merge request as inline review comments anchored to file:line **plus** a top-level summary, prefixing **every** comment with the attribution line so each one is clearly machine-generated and human-curated.

This is the write-side complement to `/check-feedback` (which only *reads* feedback). The content here is feedback you already have — typically the output of `/review` or `/code-review`, or text the user supplies. The "curated by Sal" attribution means a human signed off, so **always show the exact comments and get explicit confirmation before posting.**

## The attribution prefix

Every posted comment (inline and summary) must begin with this exact line, followed by a blank line:

```text
:robot: Generated by AI but curated by Sal

<the comment body>
```

Use this verbatim. Do not paraphrase, translate, or drop the emoji.

## Usage

```text
/post-feedback [revset|number]
```

Examples:
- `/post-feedback` → infer the PR/MR from the bookmark on `@` (or nearest ancestor with a tracked bookmark)
- `/post-feedback @-` → use the bookmark on the parent change
- `/post-feedback 1234` → post to PR/MR number `1234` directly

If the feedback itself isn't already in the conversation, ask the user for it (or run `/review` first).

## Workflow

1. **Detect forge**
   ```bash
   git remote get-url origin
   ```
   - GitHub host → use `gh`.
   - GitLab host → use `glab`.
   - Verify auth before posting:
     ```bash
     gh auth status   # or: glab auth status
     ```

2. **Resolve the PR/MR number** (same resolution as `/check-feedback`)
   - If the user passed a bare number, use it.
   - Otherwise resolve from the revset:
     ```bash
     jj log -r '<rev>' --no-graph -T 'bookmarks ++ "\n"'
     ```
     - If no bookmark on the rev, walk back to the nearest ancestor with one:
       ```bash
       jj log -r '::<rev> & bookmarks()' --no-graph -T 'change_id.short() ++ " " ++ bookmarks ++ "\n"'
       ```
     - Map the bookmark to a PR/MR:
       ```bash
       gh pr list --head <bookmark> --state all --json number,url,state,title
       # or
       glab mr list --source-branch <bookmark> --all
       ```
   - If nothing matches, report that no PR/MR exists and stop. Don't guess.

3. **Anchor inline comments to the current head**
   - Inline comments must point at lines that exist in the PR's head commit, not stale local lines.
   - Get the head SHA the PR is currently showing:
     ```bash
     gh pr view <number> --json headRefOid -q .headRefOid
     ```
   - For each inline finding, capture `path`, the line number in the head version of the file, and `side` (`RIGHT` for added/context lines, `LEFT` for removed lines). Use `start_line`+`line` for multi-line ranges.
   - If a finding's line no longer exists at head, demote it into the summary instead of anchoring it — a bad anchor makes the API reject the whole review.

4. **Apply the prefix to every comment body**
   - Prepend the attribution line + blank line to the summary body and to each inline comment body. No exceptions.

5. **Confirm before posting** (outward-facing — required)
   - Show the user the full set: the summary body and each inline comment with its file:line anchor, all with the prefix applied.
   - Get explicit go-ahead. Do not post on inference.

6. **Post — GitHub (`gh`)**

   Post inline comments and the summary as a single review (atomic; groups everything under one submission). Build a JSON payload and pipe it to `gh api` — array fields via `-f`/`-F` flags are fragile, JSON input is reliable. `{owner}`/`{repo}` are auto-substituted by `gh`.

   ```bash
   gh api --method POST repos/{owner}/{repo}/pulls/<number>/reviews --input - <<'JSON'
   {
     "commit_id": "<headRefOid>",
     "event": "COMMENT",
     "body": ":robot: Generated by AI but curated by Sal\n\n<summary text>",
     "comments": [
       {
         "path": "src/foo.ts",
         "line": 42,
         "side": "RIGHT",
         "body": ":robot: Generated by AI but curated by Sal\n\n<inline finding>"
       }
     ]
   }
   JSON
   ```

   - `event: "COMMENT"` posts feedback without an approval decision. Use `REQUEST_CHANGES` or `APPROVE` only if the user explicitly asks.
   - If the user instead wants the summary in the main conversation thread (not as the review body), post it separately:
     ```bash
     gh pr comment <number> --body ":robot: Generated by AI but curated by Sal

   <summary text>"
     ```

7. **Post — GitLab (`glab`)**
   - Summary as a note on the MR:
     ```bash
     glab mr note <iid> --message ":robot: Generated by AI but curated by Sal

   <summary text>"
     ```
   - Inline comments are positional discussions and need full diff-position context. Fetch the SHAs and post one discussion per line via the API:
     ```bash
     glab api projects/:fullpath/merge_requests/<iid> --jq '{base: .diff_refs.base_sha, start: .diff_refs.start_sha, head: .diff_refs.head_sha}'

     glab api --method POST projects/:fullpath/merge_requests/<iid>/discussions \
       -f 'body=:robot: Generated by AI but curated by Sal

   <inline finding>' \
       -f 'position[position_type]=text' \
       -f 'position[base_sha]=<base_sha>' \
       -f 'position[start_sha]=<start_sha>' \
       -f 'position[head_sha]=<head_sha>' \
       -f 'position[new_path]=src/foo.ts' \
       -f 'position[new_line]=42'
     ```

8. **Report what was posted**
   - Link to the posted review/comment(s). For GitHub the review URL comes back as `html_url` in the API response; otherwise point at `gh pr view <number> --web`.
   - State how many inline comments landed and whether any findings were demoted to the summary (per step 3).

## Notes

- **The prefix is non-negotiable and goes on every comment** — inline and summary alike. If you can't apply it to a given comment, don't post that comment.
- **Confirm before posting.** This writes to a PR other people see; treat it like sending mail. Approval to post one batch does not extend to a later one.
- A single bad inline anchor (line that doesn't exist at head) rejects the *entire* review on GitHub. Verify anchors against `headRefOid`, and demote anything stale into the summary.
- Multi-line GitHub comments: include both `start_line` and `line` (and matching `side`/`start_side`).
- Use the review API (`/pulls/<n>/reviews`) to post inline + summary atomically rather than firing many `/pulls/<n>/comments` calls.
- Cite **Jujutsu change IDs** for local changes and **PR/MR URLs** for posted feedback so the user can click straight to either side.
- Git is a fallback only if `jj` is unavailable in the repo.
